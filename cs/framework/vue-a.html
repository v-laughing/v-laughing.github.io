基本概念 ​

模板：将 DOM 绑定至底层 Vue 实例的数据。

指令：决定如何处理DOM元素。

基本实现 ​

view => model：view上对dom结点的操作会触发相应的事件，在事件回调函数中实现对model值的赋值。 model => view：model上对对象的赋值会触发对象的访问器事件，因而可以在此实现对view的更新。

html
<div id="app">
    <input type="text" v-model="text">
    {{ text }}
</div>
<script>
var vm = new Vue({
	el: 'app',
	data: {
		text: 'hello world'
  	}
})
</script>

数据初始化：解析app下的所有子节点，将所有的绑定值替换或赋值为model里的值。

数据劫持：为有v-model属性的元素添加事件监听器，实现view=>model。

订阅/发布模式：因为存在多个节点的内容对应同一个属性值，可以为每个节点设置一个订阅者对象，将这个订阅者对象加入这个属性的订阅者列表中。与此同时，属性的访问器成为发布者，从而实现model=>views。

由于数据劫持采用的是Object.defineProperty()，一些对属性的操作无法拦截，如通过下标方式修改数组数据或者给对象新增属性。

Vue3.0修复了这问题，其使用Proxy 对对象进行代理：监听的范围更广，封装更好。

变化侦听 ​

侦测数据的变化。当数据变化时，更新视图，即model=>view过程。vue主要通过[[front-end#数据绑定|数据劫持]]方式实现。主要有几点：

  • 行为拦截：通过Object.defineProperty对对象的属性的getter和setter行为进行拦截。

  • 依赖收集：收集那些使用该数据的地方，在getter中进行，将其放入一个dep对象，其中有包含依赖的数组subs。

  • 依赖抽象：对那些使用数据的地方的抽象，称为watcher，包含数据变化时的操作，如回调或视图更新。新建时会自动被添加入subs。

  • 侦测封装：实现一个Observer类，对一个对象的所有属性递归地进行侦测。

Object ​

基本实现

js
class Observer {
  constructor(obj) {
    this.obj = obj
    if (!Array.isArray(obj)) this.walk(obj);
  }
  walk(obj) {
    for (let key of Object.keys()) {
      defineReactive(obj, key, obj[key])
    }
  }
}

function defineReactive(data, key, val) {
  if (typeof val === 'object') new Observer(val);
  
  let dep = new Dep()
  Object.defineProperty(data, key, {
    enumerable: true,
    configurable: true,
    get() {
      dep.depend() // 收集依赖
      return val
    },
    set(newVal) {
      if (val === newVal) return;
      val = newVal
      dep.notify() // 视图更新
    }
  })
}

问题

采用Object.defineProperty，对对象地添加或删除属性行为无法进行拦截。

数组 ​

由于对象增删属性的行为无法拦截，因此对于数组这类频繁进行增删元素的对象而言，并不适合直接对其属性，即索引进行侦测。因而采取对数组方法的拦截方式，即对原生方法的覆盖。

两者不能共存，因为原生数组方法的内部含有对属性的访问与赋值。同时存在时访问器将被触发两次。

由于侦测是在整个数组上的，而非单个属性，因此Dep对象放在Observer中。

对于数组内的对象或数组（不是侦听索引），同样需要进行侦听。

问题

显然，基于上述设计，无法对数组的属性变化进行侦听，如

js
this.list[0] = 1
this.list.length = 0

缺陷处理 ​

Vue采用拓展api的方式（本质也是拦截）解决上述的问题，即

js
vm.$set(target, key, val)
vm.$delete(target, key)

对数组的处理实际上通过新的splice方法代理。

vue3 ​

Watcher ​

每个watcher对应一个组件，可观察多个数据，因此Dep和watcher是多对多关系。

新建watcher时，其被自动添加入对应的subs列表中。

js
class Watcher {
  constructor(vm, expOrFn, cb) {
    this.vm = vm
    this.deps = [] // 可观察多个数据
    this.depIds = new Set()
    // 执行this.getter()，会读取所有属性的值，函数内的或'a.b.c'
    this.getter = isFunction(expOrFn) ? expOrFn : parsePath(expOrFn)
    this.cb = cb
    this.value = this.get()
  }
  get() { // 读取属性值
    window.target = this
    let value = this.getter.call(this.vm, this.vm)
    window.target = undefined
    return value
  }
  update() { // 执行回调
    const oldValue = this.value
    this.value = this.get()
    this.cb.call(this.vm, this.value, oldValue)
  }
}

apivm.$watch内部通过新建一个watcher对象实现。

Virtual DOM ​

虚拟DOM：用 JS 对象表示 DOM ，两者可以相互转换。

通过JS对象的缓存操作，可以减少对DOM的操作次数。大致过程：

  1. 将DOM用JS对象表示。主要是存储节点的类型，属性，子节点。
  2. DOM状态变更时，重新构造一棵新的对象树。用diff算法找出新旧两树的不同。
  3. 根据得到的不同对DOM进行相应的操作。

vue方案 ​

可以对每个使用了数据的节点绑定一个watcher，进而实现节点与数据的绑定，但这样做内存开销很大。

vue通过引入虚拟DOM，采取了一种折中的方案：每个组件对应一个watcher。当数据发生变化时，所有依赖该数据的组件会通过虚拟DOM去进行比对，进而更新渲染那些需要变化的节点。

diff算法 ​

虚拟DOM的应用关键在于diff算法，其找出新旧两颗虚拟DOM树差异。vue采取的方式为同层比较。

模板编译 ​

将模板编译，最终解析为一个渲染函数。

主要有三部分内容：

  • 将模板解析为抽象语法树AST。

  • 遍历AST标记静态节点。

  • 使用AST生成代码字符串，作为渲染函数的参数。

下面是一个例子。

模板

html
<p title="Berwin" @click="c">hello {{name}}</p>

生成的代码字符串

js
with (this) {
  return _c( // createElement
    'p',
    {
      attrs: { "title": "Berwin" },
      on: { "click": c }
    },
    [_v("hello" + _s(name))] // _v: createTextVNode _s: toString
  )
}

执行渲染函数，生成VNode。

当解析的节点为组件节点时，会将其标签属性上解析出的数据当作参数传递给子组件。

异步更新 ​

vue数据的变更不会立即执行DOM的更改，这些变更状态（watcher）被放入一个更新队列中。这种策略可以减少对DOM的操作。大致过程为：

  1. 一开始，数据变更时，通过nextTick将对应的watcher放入队列queue，queue被放入更新队列callbacks中，同时开启一个宏或微任务去执行更新队列。

直到该任务执行之前的这段时间，发生的数据变更被去重地压入队列，但不会再开启宏或微任务。

  1. api调用nextTick时会将其回调函数压入callbacks。

  2. 执行watcher完成时，此时原生DOM状态发生更改，但未在页面显示。

  3. 当当前事件循环结束时（即微队列的任务全部被执行），浏览器渲染页面，此时所有状态变更在页面上生效。

nextTick ​

从上面可以得知，nextTick的回调函数发生在DOM更新之后，既有：

js
vm.message = 'new message' // 更改数据
vm.$el.textContent === 'new message' // false
Vue.nextTick(function () {
  vm.$el.textContent === 'new message' // true
})

显然，根据其原理，可以使nextTick替换成setTimeout，从而达到一样的效果。

宏/微任务 ​

nextTick默认采用微任务，如果不行，则退为宏任务。具体的顺序为：

promise.then -> MutationObserver -> setImmediate -> setTimeout

生命周期 ​

初始化 ​

vue示例创建时，会进行一系列初始化，主要是：

js
// 合并options
vm.$options = mergeOptions() 
// 向实例挂载属性，如$parent, $_wacther等
initLifecycle(vm) 
// 将在父组件使用的v-on注册的事件添加到子组件的事件系统
initEvents(vm) 
initRender(vm) // render 相关的属性，_c, $attrs
callHook(vm, 'beforeCreate') 
// 在data/props前初始化inject, 获取父组件的provide值并初始化
initInjections(vm) 
initState(vm)
initProvide(vm) // 在data/props后初始化provide
callHook(vm, 'created')
if (vm.$options.el) {
vm.$mount(vm.$options.el)
}

以 $ 开头的属性是提供给用户使用的，以 _ 开头的属性是提供给内部使用的。

inject&provide ​

祖先组件通过provide注入数据，子孙组件可以通过inject这些数据。

inject在data前初始化，说明可以在data数据中使用inject的数据。

state ​

初始化主要的数据属性：props -> methods -> data -> computed -> watch。

props 将父组件的props数据传给子组件规格化的props属性，将最终的数据保存在vm._props中，通过代理实现vm.x访问vm._props.x。

methods 检查是否合法，遍历其属性，赋值给vm。

computed 主要特点是有数据缓存(存在watcher中)，使用watcher来观察其中的数据变化。

这样的实现有个问题：如果计算属性中的状态发生了变化，但计算属性的值并没有变，此时仍然会重新渲染组件。

改进的方案是：计算属性的watcher判断最终值是否改变，来决定是否通知组件的watcher。

生命钩子 ​

组件从创建到销毁的的过程，主要包括：开始创建、初始化数据、编译模板、挂载Dom→渲染、更新→渲染、销毁。

Vue提供了组件在生命周期各个阶段的钩子函数，以此在各个阶段实现一些操作。

beforeCreate： 实例初始化前被触发，访问不到data数据。

created：创建实例，完成数据初始化后。可访问 data、methods，但访问不到 $el 属性。

一般在此做一些页面初始化的工作，比如ajax 请求数据。

beforeMount：模板解析完成。

mounted：组件元素创建完成，准备挂载。

并不一定添加到了DOM，可以用nextTick来保证这一点。

beforeUpdate： 数据更新将要对DOM做一些更改时。

对可能会被移除的元素做一些操作，比如移除事件监听器。

updated：在DOM的更改及渲染完毕后。

beforeDestroy：组件即将被销毁并且从DOM上移除时。

销毁定时器、解绑全局事件等。

destroyed：组件被销毁后。

实例中的所有东西都会解除绑定，包括事件监听器，所有的子实例。

使用 keep-alive时，还有两个钩子函数：分别是 activated 和 deactivated 。用 keep-alive 包裹的组件在切换时不会进行销毁，而是缓存到内存中并执行 deactivated 钩子函数，命中缓存渲染后会执行 actived 钩子函数。

父子组件 ​

父子间的生命周期执行顺序：对于某个过程，先执行父元素的前置钩子，再执行子元素的整套钩子，最后执行父元素的钩子。

  • 加载渲染过程
html
父beforeCreate->父created->父beforeMount->子beforeCreate->子created->子beforeMount->子mounted->父mounted
  • 子组件更新过程
html
父beforeUpdate->子beforeUpdate->子updated->父updated
  • 父组件更新过程
html
父beforeUpdate->父updated
  • 销毁过程
html
父beforeDestroy->子beforeDestroy->子destroyed->父destroyed
