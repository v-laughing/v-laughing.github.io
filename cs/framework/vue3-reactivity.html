基础 ​

目的：监听某些值的变化，当其发生改变时，依赖于该值的其他值能相应的更新。

如下所示，当count变化时，我们希望模板能够相应的更新。

vue
<template>
  <button @click="count++">{{ count }}</button>
</template>
<script setup>
  const count = ref(0)
</script>

为了更加一般的情况，可以将这些依赖于监听值的行为抽象为一个函数，称为依赖函数。这样，当监听值变化时，我们需要执行该函数。

这样子，我们确定了该目的的基本要素，即：要监听的数据和要更新的依赖。

要监听的数据是可以被确定的，比如集中存储于某个对象（vue2中的data对象），或者单独指定某个数据（vue3中的ref和reactive）。因此，接下来要解决的有两个问题：如何收集依赖、如何在监听的数据变化时执行依赖。

依赖收集与执行 ​

所谓依赖即是使用到被监听数据的地方。

依赖包含了对数据的访问，通过拦截或重写数据的访问过程，可以在该过程实现对依赖的收集。

当被监听数据发生变化时，此时应该执行所有依赖，因此数据的更改行为也需要被拦截或重写。

  1. JS对象中，Object.defineProperty()方法可以重写对象的某个键的访问器属性，进而对键的访问与更改的拦截操作。
js
let data = {'key': 1}
Object.defineProperty(data, 'key', {
  get() { // 依赖收集
    return 2
  },
  set(newVal) {} // 依赖执行
})
  1. JS的 Proxy 对象也可以提供对对象属性的读取与更改的拦截操作。代理对象可认为是一类具有与Object相同操作行为且可定制的容器，通过定制容器内的行为进而控制原始对象的行为。
js
let obj = { a: 1 }

const p = new Proxy(obj, {
  get(target, key, value) {
    console.log('get')
    return target[key]
  },

  set(target, key, value) {
    target[key] = value;
  }
})

p.b = 2

基于某些原因，vue2中选择了第一种实现，vue3选择了第二种。

响应式系统 ​

现在，考虑要跟踪的数据是Object，准确的说，跟踪Object内的属性，而不是Object本身。

js
let obj = { a: 1}
obj.a = 2 // 能被响应
obj = 2 // 不能被响应

组织依赖 ​

要监听的数据target可以是多个，每个数据内的属性 key 也可以有多个依赖。可以通过一个WeakMap将所有数据及其收集的依赖关联起来。

选择WeakMap而不是Map的原因在于：WeakMap 对 key 是弱引用，不计入垃圾回收器对key的引用计数。当外部对 key 的计数归零时，垃圾回收器会将其清除，此时在WeakMap中将访问不到 key 的值。

基本设计 ​

有了前面的内容，可以设计个简单的依赖收集与执行系统。

js

const bucket = new WeakMap()
const data = {}

const obj = new Proxy(data, {
  get(target, key) {
    track(target, key)
    return target[key]
  },
  set(target, key, newVal) {
    target[key] = newVal
    trigger(target, key)
  }
})

function track(target, key) {
  if (!activeEffect) return
  let depsMap = bucket.get(target)
  if (!depsMap) {
    bucket.set(target, (depsMap = new Map()))
  }
  let deps = depsMap.get(key)
  if (!deps) {
    depsMap.set(key, (deps = new Set()))
  }
  deps.add(activeEffect) // activeEffect 即是当前的依赖函数
}

function trigger(target, key) {
  const depsMap = bucket.get(target)
  if (!depsMap) return
  const effects = depsMap.get(key)
  effects && effects.forEach(fn => fn())
}

现在的问题是activeEffect该如何准确地与key对应。考虑下面的代码，我们可以这样来定义activeEffect。

js
function log(){
  console.log(data.msg)
}
setTimeout(() => {data.msg = 'hello'})

activeEffect = log // 依赖函数的执行会触发依赖的收集，在此之前将该函数赋值给activeEffect
log()

更一般地，将下面的最后两行抽象为effect函数，即：

js
let activeEffect // 全局变量
function effect(fn) { // fn 为依赖函数
  activeEffect = fn
  fn()
}

对于任何的依赖函数，都应该执行一遍effect函数，已使得依赖被正确地收集。

边界问题 ​

无效的依赖函数执行 ​

考虑下面一种情况，当obj.ok为由true变为false时，此时obj.text的变化是不需要执行该依赖的。

js
effect(()=>{
  document.body.innerText = obj.ok ? obj.text : 'not'
})

解决该问题的方法是，在每次依赖函数执行前，将它从包含它的依赖集合中删除，执行时该依赖被再度收集。

为此，需要记录依赖函数存在于哪些集合中。每个依赖函数effectFn的deps属性保存包含该依赖的集合。

js
function effect(fn) {
  const effectFn = () => {
    cleanup(effectFn)  // 新增
    activeEffect = effectFn
    fn()
  }
  effectFn.deps = [] // 依赖函数存在于哪些依赖集合中
  effectFn()
}

function cleanup(effectFn) {
  for (let i = 0; i < effectFn.deps.length; i++) {
    const deps = effectFn.deps[i]
    deps.delete(effectFn)
  }
  effectFn.deps.length = 0
}

function track(target, key) {
  // ...
  activeEffect.deps.push(deps) // 新增
}

function trigger(target, key) {
  // ...
  const effectsToRun = new Set(effects)  // 不能直接执行，否则会产生循环
  effectsToRun.forEach(effectFn => effectFn()) 
}

effect嵌套 ​

vue组件的更新策略是，当组件依赖的某个数据发生变化时，整个组件将「重新渲染」。可以认为，一个组件就是一个依赖函数effect，组件可以嵌套，因此effect也可。

js
effect(function fn1(){
  effect(function fn2(){ console.log(obj.fn2) })
  console.log(obj.fn1)
})

如上，根据前面的设计，最终obj.fn1关联到的依赖将是fn2，这是不对的。问题在于，内层副作用函数的执行会覆写了全局变量activeEffect的值。

这可以引入依赖函数栈effectStack来解决。

js
let activeEffect
const effectStack = []
function effect(fn) {
  const effectFn = () => {
    cleanup(effectFn)
    activeEffect = effectFn
    effectStack.push(effectFn) // 入栈
    fn()
    // 依赖函数执行完后，弹出栈，并把 activeEffect 还原为之前的值
    effectStack.pop()
    activeEffect = effectStack[effectStack.length - 1]
  }
  // ...
}

无限递归 ​

当依赖函数中同时对一个属性的读取与更新操作时，将产生无限递归。因为在依赖函数执行过程中，属性更新使得该依赖再次被调用。

js
effect(() => {
  obj.foo = obj.foo + 1
})

显然，在依赖函数执行过程中，应阻止其在trigger函数中被调用。通过：

js
function trigger(target, key) {
  // ...
  effects && effects.forEach(effectFn => {
    // 如果 trigger 触发执行的依赖函数与当前正在执行的依赖函数相同，则不触发执行
    if (effectFn !== activeEffect) {  // 新增
      effectsToRun.add(effectFn)
    }
  })
  // ...
}

功能拓展 ​

可调度性 ​

当属性值发生变化时，我们希望可以控制依赖函数执行的时机、次数以及方式。

js
effect(
  () => { console.log(obj.foo) },
  // options
  {
    scheduler(fn) {
      // fn，即依赖回调，在这里可以自己实现对它的执行控制
    }
  }
)

function effect(fn, options = {}) {
  // ...
  effectFn.options = options  // 新增
  // ...
}

function trigger(target, key) {
  // ...
  effectsToRun.forEach(effectFn => {
    if (effectFn.options.scheduler) {
      effectFn.options.scheduler(effectFn)
    } else {
      effectFn()
    }
  })
}

基于调度器，很容易实现一个「延时集中更新」的操作，具体的说，是在一个事件循环中集中更新一次。

js
const jobQueue = new Set() // 便于去重
const p = Promise.resolve()
let isFlushing = false

function flushJob() {
  if (isFlushing) return
  isFlushing = true
  p.then(() => {
    jobQueue.forEach(job => job())
  }).finally(() => {
    isFlushing = false
  })
}

effect(() => { console.log(obj.foo)}, 
  {
  scheduler(fn) {
    jobQueue.add(fn)
    flushJob()
  }
})

计算属性 ​

计算属性有以下几点特征：

  • 依赖于其他被监听的数据，当依赖的响应式数据更新时，其读取值能跟着变化。
  • 当读取该值时，才进行计算，且能够对值进行缓存。（当依赖数据不变时，不进行计算）
  • 本身是个响应式数据，被监听。

上述实现的effect函数，会在依赖变化时执行依赖函数。如果将该依赖函数的返回值定义为一个「计算属性」，就可以实现第一个目的。

为了实现第二点，可以引入lazy选项，避免调用effect时依赖函数的立即执行，同时effect返回依赖函数，用于手动执行。对值的缓存可以用一dirty变量解决，关键在于使得其在scheduler中被设置为true。（scheduler在依赖数据变化时执行）。

第三点，计算属性应该为一对象，且需要对其进行手动监听。

js
function effect(fn, options = {}) {
  const effectFn = () => {
    // ...
    const res = fn()  //
    // ...
    return res  // 依赖函数的返回值
  }
  if (!options.lazy) { // 不立即执行
    effectFn()
  }
  return effectFn // 返回依赖函数，执行时重新计算值
}

function computed(getter) {
  let value
  let dirty = true

  const effectFn = effect(getter, {
    lazy: true,
    scheduler() {
      if (!dirty) {
        dirty = true
        // 当计算属性依赖的响应式数据变化时，手动调用 trigger 函数触发响应
        trigger(obj, 'value')
      }
    }
  })

  const obj = {
    get value() {
      if (dirty) {
        value = effectFn()
        dirty = false
      }
      // 当读取 value 时，手动调用 track 函数进行追踪
      track(obj, 'value')
      return value
    }
  }

  return obj
}

watch ​

watch的特点有：

  • 监听某些数据，在其发生变化时，手动执行某些行为。
  • 能够拿到监听数据的新旧值。

第一点利用scheduler机制可以轻松完成。

第二点，关键在于对上一次计算结果的缓存。利用lazy特性，可以手动执行effect函数获得依赖数据最开始的值。

js
function watch(source, cb) {
  let getter
  if (typeof source === 'function') { // 可以是函数
    getter = source
  } else {
    getter = () => traverse(source) // 遍历读取数据
  }
  let oldValue, newValue
  const effectFn = effect(
    () => getter(),
    {
      lazy: true,
      scheduler() {
        newValue = effectFn()
        cb(newValue, oldValue)
        oldValue = newValue
      }
    }
  )
  // 手动调用副作用函数，拿到的值就是旧值
  oldValue = effectFn()
}

function traverse(value, seen = new Set()) {
  if (typeof value !== 'object' || value === null || seen.has(value)) return
  seen.add(value)
  for (const k in value) { // 递归遍历
    traverse(value[k], seen)
  }
  return value // 返回本身
}

更近一步，可以通过参数的形式指定watch回调函数的执行时机。例如是否立即执行immediate或在一个周期内的执行时期flush。

js
function watch(source, cb, options = {}) {
  const job = () => {
    newValue = effectFn()
    cb(newValue, oldValue)
    oldValue = newValue
  }
  const effectFn = effect(
    // 执行 getter
    () => getter(),
    {
      lazy: true,
      scheduler: job
    }
  )
  if (options.immediate) {
    job()
  } else {
    oldValue = effectFn()
  }
}

过期问题：对于像下面的情况，obj的变化先后触发了A, B两个请求，但A请求慢于B返回，此时数据将被更新为A的返回。

js
let finalData
watch(obj, async () => {
  const res = await fetch('/path/to/request')
  finalData = res
})

解决问题的关键在于，后面触发的变化（回调函数执行时），应该使上一个的变化无效，或者说，过期。

js
// 每个回调函数有一个过期标志，下一个触发的回调将该值置为真
watch(obj, async (newValue, oldValue, onInvalidate) => {
  let expired = false 
  onInvalidate(() => {
    expired = true
  })
  const res = await fetch('/path/to/request')
  if (!expired) {
    finalData = res
  }
})

function watch(source, cb, options = {}) {
  let cleanup
  function onInvalidate(fn) {
    cleanup = fn
  }
  const job = () => { // 依赖变化时执行，执行前将上个回调过期
    newValue = effectFn()
    if (cleanup) {
      cleanup()
    }
    cb(newValue, oldValue, onInvalidate)
    oldValue = newValue
  }
}

代理Object的其他细节 ​

从上面了解到，要想跟踪数据响应，应拦截对象的读取和更新操作，在读取操作中收集依赖，在更新操作中执行依赖。

对于Object，有如下的读取操作。

  • obj.foo：可通过内部方法 get 拦截。
  • key in obj：可通过内部方法 has 拦截。
  • for (const key in obj)：可通过内部方法 ownKeys 拦截，该函数返回只属于对象自身拥有的键。
js
const obj = { foo: 1 }
const ITERATE_KEY = Symbol()

const p = new Proxy(obj, {
  has(target, key) {
    track(target, key)
    return Reflect.has(target, key)
  }，
  ownKeys(target) {
    track(target, ITERATE_KEY) // 没有具体的key
    return Reflect.ownKeys(target)
  }
})

值得注意的是，对象的for in操作仅仅是遍历对象的键，因此仅仅当对象的键新增或减少时才需要重新执行依赖。

Object的更改操作还有delete，其行为由deleteProperty方法定义。

js
const p = new Proxy(obj, {
  deleteProperty(target, key) {
    const hadKey = Object.prototype.hasOwnProperty.call(target, key)
    const res = Reflect.deleteProperty(target, key)
    if (res && hadKey) { // 只有当被删除的属性是对象自己的属性并且成功删除时，才触发更新
      trigger(target, key, 'DELETE')
    }
    return res
  }
})

其他需要考虑的细节还有：

  • 当值没有发生变化时，不需要触发响应。
  • 因原型链的设计，设置的属性不存在于对象上时，会取得其原型，并调用原型的 [[Set]] 方法，这使得原型也发生了不必要的更新响应。
  • obj的响应应该是递归的，即深响应。当然，也可以是仅浅响应shallowReactive。
  • 有些数据是只读的，此时无需对其收集依赖，并且不允许其设置值。

为了后续讲解方便，封装一个 reactive 函数，为一个原始对象创建响应式数据。

js
function reactive(obj) {
  return new Proxy(obj, {
    // ...
  })
}

深响应 ​

上面的实现中，reactive是浅响应的，这意味着下面 obj.foo.bar 的赋值不会使得依赖被执行。

js
const obj = reactive({ foo: { bar: 1 } })
effect(() => {
  console.log(obj.foo.bar) // no
})
obj.foo.bar = 2

显然，我们需要内部的对象也封装为响应式数据，这是个递归的过程。为了更好的节省资源，可以仅仅在访问内部对象时将其响应化，而不必在一开始时操作。

js
function reactive(obj) {
  return new Proxy(obj, {
    get(target, key, receiver) {  // 在 get 操作时
      if (key === 'raw') {
        return target
      }
      track(target, key)
      const res = Reflect.get(target, key, receiver)
      if (typeof res === 'object' && res != = null) {
        return reactive(res) // here
      }
      return res
    }
  })
}

代理数组 ​

数组是一个特殊的对象，数组的索引即是对象的key。因此，对存在的索引的赋值是可以监听到数组的变动的。

js
const arr = reactive(['foo'])
effect(() => {
console.log(arr[0]) // 'foo'
})
arr[0] = 'bar' // 能够触发响应

为了代理数组，首先需知道数组的操作。

  • 访问操作：arr[i]，arr.length，for...in ，for...of ，arr.find等方法。
  • 更改操作：arr[i]，arr.length，arr.push等方法。

索引与长度 ​

对数组的索引或长度赋值时，需要注意：

  • 当设置的索引值大于数组当前的长度，数组会更新 length 属性。因此需要相应的执行 lengthEffects。
  • 当设置的长度小于数组当前的长度，数组后面的元素会被清空。同样需要触发后面元素的 effects。

为了区分设置索引值是否需要引起lengthEffects的执行，可以通过type字段标志。值为ADD表示需要。

js
function createReactive(obj) {
  return new Proxy(obj, {
    set(target, key, newVal, receiver) {
      const oldVal = target[key]
      const type = Array.isArray(target)
        ? Number(key) < target.length ? 'SET' : 'ADD'
        : Object.prototype.hasOwnProperty.call(target, key) ? 'SET' : 'ADD'

      const res = Reflect.set(target, key, newVal, receiver)
      if (target === receiver.raw) {
        if (oldVal !== newVal && (oldVal === oldVal || newVal === newVal)) {
          trigger(target, key, type, newVal)
        }
      }
      return res
    },
  })
}
js
function trigger(target, key, type) {
  const depsMap = bucket.get(target)
  if (!depsMap) return

  if (type === 'ADD' && Array.isArray(target)) {
    const lengthEffects = depsMap.get('length') // 执行 lengthEffects
    lengthEffects && lengthEffects.forEach(effectFn => {
      if (effectFn !== activeEffect) {
        effectsToRun.add(effectFn)
      }
    })
  }

  if (Array.isArray(target) && key === 'length') { // 更新被长度置空的元素依赖
    depsMap.forEach((effects, key) => {
      if (key >= newVal) {
        effects.forEach(effectFn => {
          if (effectFn !== activeEffect) {
            effectsToRun.add(effectFn)
          }
        })
      }
    })
  }
}

遍历与查找 ​

对于 for...in，如上所述，对于一个普通对象，仅当添加或删除属性值时才会影响依赖。

对于数组而言，这两种情况本质上改变了 length 值。因此for...in操作的依赖可以绑定在length键上。

js
function createReactive(obj) {
  return new Proxy(obj, {
    ownKeys(target) {
      track(target, Array.isArray(target) ? 'length' : ITERATE_KEY)
      return Reflect.ownKeys(target)
    }
  })
}

对于 for...of，我们希望数组的任何改变都可以重新执行相关依赖。

for...of是一个迭代过程，遍历迭代器，每次返回一个值。其内部实现过程访问到了数组的 length 与 每个索引i，因此相关的依赖自然会与长度和索引绑定，从而监听到数组的任何变化。故无需为此添加代码。

同样地，对于许多数组的查找方法，如find，其内部实现也依赖与数组的基本语义，其依赖也无需特别跟踪。

对于includes，indexOf等方法，情况有所不同。

js
const obj = {}
const arr = reactive([obj])
console.log(arr.includes(obj)) // false

在includes方法内部，this指向是代理对象，通过遍历代理对象的每个值去与传入的参数对比。由于深响应的实现，代理对象中的obj是响应式对象，因而查找失败。

可以通过重写includes方法实现：

js
const originMethod = Array.prototype.includes
const arrayInstrumentations = {
  includes: function(...args) {
    let res = originMethod.apply(this, args) // 不能省略，为了跟踪依赖；同时查找浅响应
    if (res === false) {
      res = originMethod.apply(this.raw, args) // 查找深响应
    }
    return res
  }
}

无限循环 ​

对于某些数组方法，如push，pop等，其内部实现中，读取并设置了数组的 length 值。我们在上面中处理了这种情况，使得下面的操作不会导致无限递归。

js
effect(() => {
  arr.push(1) 
})

但下面的操作会导致无限循环。第一个依赖和第二个依赖会无限交替执行。

js
const arr = reactive([])
effect(() => {
  arr.push(1)
})
effect(() => {
  arr.push(1)
})

实际上，从使用的角度看，push仅仅是一个更改操作，而不包含读取操作——不期望有变更时执行该方法。因而可以在push中屏蔽掉对length的跟踪。

js
let shouldTrack = true

['push'].forEach(method => {
  const originMethod = Array.prototype[method]
  arrayInstrumentations[method] = function(...args) {
    shouldTrack = false // 屏蔽跟踪
    let res = originMethod.apply(this, args)
    shouldTrack = true
    return res
  }
})

代理 Set 和 Map ​

这两个对象不同于普通对象，其基本操作主要通过特定的方法。

同样的，通过一个代理对象对其操作进行拦截：

js
const s = new Set([1, 2, 3])
const p = new Proxy(s, {})
console.log(p.size) // error1 
p.delete(1) // error2

上述的代码会报错，原因在于Set或Map的操作要求对应的对象必须满足某些特性，而代理对象p并不满足。

为了解决该问题，不必实现这些特性，只需操作转发给原始set对象。

js
const p = new Proxy(s, {
  get(target, key, receiver) {
    if (key === 'size') {
      return Reflect.get(target, key, target)
    }
    return target[key].bind(target)
  }
})

同样地，为了建立对集合类型的响应，需要知道它们的访问与更新操作，在访问操作中绑定依赖，而在更新操作中执行依赖。

对于Set。

  • 访问操作：size、has()、keys()、values()、entries()、forEach()。
  • 更新操作：add()、delete()、clear()。

对于Map。

  • 访问操作：size、get()、has()、keys()、values()、entries()、forEach()。
  • 更新操作：set()、delete()、clear()。

当集合类型被更新时，所有的访问操作的依赖都应该被执行。因此可以将所有的访问依赖都绑定在一个键上，如ITERATE_KEY。

以Set对象为例，访问size时建立响应联系。

js
function createReactive(obj) {
  return new Proxy(obj, {
    get(target, key, receiver) {
      if (key === 'raw') return target // 通过代理对象获取原始对象
      if (key === 'size') {
        track(target, ITERATE_KEY) // here
        return Reflect.get(target, key, target)
      }
      return mutableInstrumentations[key]
    }
  })
}

为了在更新方法里触发依赖执行，可用重写对象的方法。以add为例：

js
const mutableInstrumentations = {
  add(key) {
    const target = this.raw
    const res = target.add(key)
    if(!target.has(key)){         // 没有时才触发 
      trigger(target, key, 'ADD')
    }
    return res
  }
  has(key){ // has 访问方法时收集依赖
    const target = this.raw
    track(target, ITERATE_KEY)
    return target.has(key)
  }
}

深响应 ​

如同Object对象一样，集合对象也应该实现深响应。

js
const obj = reactive( new Map([["key", { b: 1}]]) )
effect(() => {
  console.log(obj.get('key').b) // 应该触发
})
obj.get('key').b = 2

更具体的，深度响应要求对象内部任何可改变的数据都是响应式的，因此Map对象的key也应该是响应式的。这与Object不同，因为其key只能是String或Symbol类型，是不可变的。

js
const obj = reactive( new Map([[{}, 1]]) )
effect(() => {
  console.log( [...a.keys()] ) // 应该触发
})
obj.forEach((v, k)=>{
  k.a = 1 // 改变key
})

因此，在数据被访问到时，如果是对象，应该将其转为响应式对象。Map对象的key在forEach()，keys(), for of中均可被访问到，在这些过程中均需要作深度绑定。

js
const wrap = (val) => typeof val === 'object' ? reactive(val) : val
const mutableInstrumentations = {
  get(key) {
    const target = this.raw
    trigger(target, key, 'GET')
    const res = target.get(key)
    return wrap(res) 
  }
  forEach(callback) {
    const target = this.raw
    track(target, ITERATE_KEY)
    target.forEach((v, k) => {
      callback(wrap(v), wrap(k), this)
    })
  }
}

数据污染 ​

在下面的例子中，可以看到，通过对原始数据的操作同样可以达到响应式的行为。

js
const m = new Map()
const p1 = reactive(m)
const p2 = reactive(new Map())
p1.set('p2', p2)

effect(() => {
  console.log(m.get('p2').size)
})
m.get('p2').set('foo', 1)

对原始数据的操作会引起其他副作用，这会使得程序更加复杂且不符合使用者预期。造成这个的原因在于：在set拦截器中直接将value赋值给了原始数据的key。

因此，为了使得原始数据更加「原始」，需要对value进行脱响应。

js
const mutableInstrumentations = {
  set(key, value) {
    const target = this.raw
    const rawValue = value.raw || value
    target.set(key, rawValue)
    trigger(target, key)
  }
}

原始值的响应式方案 ​

原始值指的是 Boolean、Number、String、undefined 和 null 等类型的值，无法直接的拦截它们的赋值过程，如let a = 'aa'。可以通过一个对象去包裹该原始值，监听对象从而间接监听原始值。 为了区分原始值的包裹对象与非原始值的响应式数据，添加了一个不可枚举且不可写的属性 __v_isRef。

js
function ref(val) {
  const wrapper = {
    value: val
  }
  Object.defineProperty(wrapper, '__v_isRef', { value: true })
  return reactive(wrapper)
}

响应丢失问题 ​

当用展开符...对响应数据操作时，得到的新对象将失去响应性。

js
const obj = reactive({ foo: 1, bar: 2 })
const newObj = {
  ...obj
}
effect(() => {
  console.log(newObj.foo)
})
obj.foo = 100

为了使newObj具有与obj关联的响应性，可以将newObj中的属性访问与设置操作转移到obj中。

js
const newObj = {
  foo: {
    get value() {
      return obj.foo
    }
    set value(val){
      obj[foo] = val
    }
  }
}

通过进一步的抽象有：

js
function toRef(obj, key) {
  const wrapper = {
    get value() {
      return obj[key]
    },
    set value(val) {
      obj[key] = val
    }
  }

  Object.defineProperty(wrapper, '__v_isRef', { value: true })
  return wrapper
}

function toRefs(obj) {
  const ret = {}
  for (const key in obj) {
    ret[key] = toRef(obj, key)
  }
  return ret
}

const newObj = {
  ...toRefs(obj)
}

自动脱ref ​

当我们对newObj的属性进行操作时，需要访问 foo.value，而不是 foo，这不是很方便。一个解决方法是，通过一个对newObj的代理，当访问foo时直接返回foo.value。

js
function proxyRefs(target) {
  return new Proxy(target, {
    get(target, key, receiver) {
      const value = Reflect.get(target, key, receiver)
      return value.__v_isRef ? value.value : value
    }
    set(){...}
  })
}
const newObj = proxyRefs({ ...toRefs(obj) })

Vue在中一些场景自动实现proxyRefs的代理：

js
const count = ref(0)
const obj = reactive({ count }) // reactive( proxyRefs({ count }) )
obj.count // 0
