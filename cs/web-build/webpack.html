Webpack 基本原理 ​

核心思想 ​

为了解决传统开发方式存在的全局作用域污染、依赖管理困难、请求过多等问题，webpack 应运而生。它的核心思想是：模块化打包。

  • 模块化：可以在 JS 文件里直接使用 import 或 require 来引入其他文件。
  • 打包：将项目里成百上千个文件（JS, CSS, 图片）压缩合并成一个或几个文件（通常叫 bundle.js）。

浏览器只需要加载这个 bundle.js，就能拥有所有的功能。

构建流程 (The Build Lifecycle) ​

Webpack 5 的构建流程主要分为三个阶段：初始化阶段、构建阶段、生成阶段。

第一阶段：初始化 (Initialization) ​

  1. 读取与合并参数：从配置文件（webpack.config.js）和 Shell 命令行参数中读取配置，合并得到最终的参数对象。
  2. 实例化 Compiler：用上一步得到的参数初始化 Compiler 对象。它是 Webpack 的单例核心，掌握整个构建周期的环境信息。
  3. 加载插件 (Load Plugins)：遍历配置中的 plugins 数组，调用插件的 apply 方法，让插件监听 Compiler 里的钩子。
  4. 环境准备：查找配置的 entry 入口，开始启动构建。

第二阶段：构建/编译 (Building / Make) ​

这是最耗时的阶段，Webpack 会从入口文件开始，一层层递归解析。

  1. 确定入口：根据配置的 entry 找出所有的入口文件。
  2. 编译模块 (Compile Module)：
    • Resolve（解析路径）：根据 import / require 语句，结合 resolve 配置（别名、后缀补全），找到文件在硬盘上的绝对路径。
    • Run Loaders（转换代码）：根据文件名匹配 rules 中的 Loader。Loader 执行顺序是从右到左（或从下到上）。例如：scss-loader -> css-loader -> style-loader。将非 JS 文件转换为 Webpack 能理解的 JS 字符串。
    • Parse（解析语法）：将转换后的 JS 代码解析为 AST (抽象语法树)。
    • Analyze Dependencies（分析依赖）：遍历 AST，寻找 import、require 等语句，收集该模块依赖了哪些其他模块。
  3. 递归编译：根据上一步分析出的依赖，递归地重复“解析 -> 转换 -> AST -> 分析”的过程。
  4. 构建模块图 (Module Graph)：当所有模块都遍历完成后，Webpack 就建立了一个完整的依赖关系图，包含了所有模块及其映射关系。

第三阶段：生成/封装 (Seal & Emit) ​

在这个阶段，Webpack 会根据依赖图生成最终的资源文件。

  1. 封装 (Seal)：这是优化的关键步骤。Webpack 会根据 entry 和 splitChunks（代码分割）策略，将多个 Module 合并成一个或多个 Chunk（代码块）。
  2. 优化 (Optimization)：执行各种优化策略，例如：
    • Tree Shaking：标记未使用的导出代码。
    • Minification：压缩代码（Terser）。
    • Module Concatenation：作用域提升（Scope Hoisting）。
  3. 生成 Assets：将 Chunk 转换成最终的文件名（包含 Hash）和文件内容，生成 assets 对象。
  4. 输出 (Emit)：这是最后修改文件的机会。Webpack 触发 emit 钩子，将 assets 写入到 output.path 指定的目录（通常是 dist 文件夹）。

核心概念 ​

  • Entry：编译的入口。
  • Module：模块，在 Webpack 里一切皆模块，一个模块对应一个文件。Webpack 会从配置的 Entry 开始递归找出所有依赖的模块。
  • Loader：根据文件名匹配 rules 中的 Loader。将非 JS 文件转换为 Webpack 能理解的 JS 字符串。
  • Chunk：代码块，一个 Chunk 由多个模块组合而成，用于代码合并与分割。
  • Output：输出结果。
  • Plugin：扩展插件，在 Webpack 构建流程中的特定时机注入扩展逻辑来改变构建结果或做你想要的事情。

最简 Webpack 演示 ​

src/utils.js (工具函数模块)

javascript
export function add(a, b) {
    return a + b;
}

src/main.js (入口文件)

javascript
import { add } from './utils.js';
// 引入 CSS（虽然 JS 不能直接运行 CSS，但 Webpack 可以处理）
import './style.css';
import logo from './logo.png';

console.log('1 + 2 = ', add(1, 2));

const img = new Image();
img.src = logo;
document.body.appendChild(img);

src/style.css

css
body {
    background-color: #f0f0f0;
}

安装与配置 Webpack ​

bash
npm install webpack webpack-cli css-loader style-loader file-loader -D

创建 webpack.config.js：

javascript
const path = require('path');

module.exports = {
    entry: './src/main.js',
    
    output: {
        filename: 'bundle.js', // 打包后的文件名
        path: path.resolve(__dirname, 'dist'), // 打包后的目录
    },
    
    module: {
        rules: [
            {
                test: /\.css$/, // 如果文件名以 .css 结尾
                use: [
                    'style-loader', // 第2步：把 CSS 内容放到 HTML 的 <style> 标签里
                    'css-loader'    // 第1步：把 CSS 文件读懂，转换成 JS 字符串
                ]
                // 注意：Loader 的执行顺序是从下往上（或从右往左）的
            },
            {
                test: /\.(png|jpg|gif)$/, // 遇到图片文件
                use: [
                    'file-loader' // 把图片移动到 dist 目录，并返回新路径
                ]
            }
        ]
    }
};

运行打包 ​

在终端执行：

bash
npx webpack

具体过程 ​

  1. Webpack 启动，读取 webpack.config.js。

  2. 找到入口 src/main.js。

  3. 分析 main.js，发现它引用了 utils.js、style.css 和 logo.png。

  4. 读取 utils.js，它是 JS，直接合并。

  5. 读取 style.css，发现它是 CSS，触发配置里的 css-loader 和 style-loader。

    • css-loader 把 CSS 内容变成了 JS 字符串。
      javascript
      // 模拟 css-loader 的输出结果
export default "body {\n    background-color: #f0f0f0;\n}";
    • style-loader 接收 CSS 字符串，生成将样式插入页面的代码：
      javascript
      // 模拟 style-loader 的输出结果
const style = document.createElement('style');
style.innerHTML = "body {\n    background-color: #f0f0f0;\n}";
document.head.appendChild(style);

  6. 读取 logo.png，发现它是图片，触发 file-loader。

    • file-loader 把图片复制到 dist/hash_logo.png。
    • 返回新图片的 URL 字符串。

  7. 最终，所有代码被封装到一个 dist/bundle.js 文件中。它的内部结构大致如下（简化版）：

    javascript
    // dist/bundle.js (简化模拟)
(function(modules) { // 1. 立即执行函数，接收模块对象
    var installedModules = {}; // 缓存

    // 2. require 函数 (Webpack 自己的模块加载器)：加载执行模块并缓存。
    function __webpack_require__(moduleId) {
        if(installedModules[moduleId]) return installedModules[moduleId].exports;
        
        var module = installedModules[moduleId] = { exports: {} };
        // 执行模块代码
        modules[moduleId].call(module.exports, module, module.exports, __webpack_require__);
        return module.exports;
    }

    // 3. 加载入口模块
    return __webpack_require__("./src/main.js");
})({
    // 4. 模块映射表：路径 -> 函数
    "./src/main.js": function(module, exports, __webpack_require__) {
        var utils = __webpack_require__("./src/utils.js");
        __webpack_require__("./src/style.css");
        var logo = __webpack_require__("./src/logo.png"); // 1. 获取图片路径
        
        console.log('1 + 2 = ', utils.add(1, 2));
        
        // 2. 使用图片
        const img = new Image();
        img.src = logo;
        document.body.appendChild(img);
    },
    "./src/utils.js": function(module, exports) {
        exports.add = function(a, b) { return a + b; };
    },
    "./src/style.css": function(module, exports) {
        // style-loader 生成的代码在这里执行
        const style = document.createElement('style');
        style.innerHTML = "body {\n    background-color: #f0f0f0;\n}";
        document.head.appendChild(style);
    },
    "./src/logo.png": function(module) {
         // file-loader 返回处理后的图片路径
        module.exports = "dist/hash_logo.png";
    }
});

最后，只需要在 HTML 里引入 dist/bundle.js 文件即可：

html
<script src="dist/bundle.js"></script>

Tree Shaking ​

Tree Shaking 是一种通过静态分析消除未引用代码（Dead Code）的技术。其核心目的是减小打包体积。

核心原理 ​

Tree Shaking 的实现主要依赖于 ES Module (ESM) 的静态结构特性。

  1. 静态分析：ESM 的导入导出语句（import/export）必须位于模块顶层，且路径必须是字符串常量。构建工具可以在编译阶段确定模块的依赖关系，而无需运行代码。
  2. 标记未引用代码：分析模块依赖图，标记出哪些导出值被使用了，哪些未被使用。
  3. 移除代码：在压缩阶段，将标记为未引用的代码从最终产物中剔除。

注意：CommonJS 是动态加载的，无法在编译阶段确定依赖关系，因此不支持 Tree Shaking。

使用方式 ​

在 Webpack 中，启动 Tree Shaking 功能必须同时满足三个条件：

  • 使用 ESM 规范编写模块代码；
  • 配置 optimization.usedExports 为 true，启动标记功能；
  • 启动代码优化功能，可以通过如下方式实现：
    • 配置 mode = production；
    • 配置 optimization.minimize = true ；
    • 提供 optimization.minimizer 数组。

副作用 ​

一般指函数调用或模块导入除了返回值之外，还修改了外部状态。

如果代码可能具有副作用，如赋值语句，函数调用，此时构建工具不会将其移除。

可以通过注释 /*#__PURE__*/ 来告诉构建工具该函数调用或模块导入没有副作用。

javascript
const foo = add(a, b) => {
    return a + b;
}
foo(1,2) // 会被保留
/*#__PURE__*/ foo(3,4) // 会被移除

禁止 Babel 转译模块导入导出语句 ​

Babel 通常用来将高版本的 JS 代码等价转译为兼容性更佳的低版本代码，以此兼容旧版本浏览器的代码。

Babel 可以将 import/export 风格的 ESM 语句等价转译为 CommonJS 风格的模块化语句，此时被转译的代码将进行静态分析，从而导致 Tree Shaking 功能失效。

为了避免这种情况，可以将 babel-preset-env 的 modules 配置项设置为 false，关闭模块导入导出语句的转译。

javascript
{
    test: /\.js$/,
    use: {
        loader: 'babel-loader',
        options: {
            presets: [['@babel/preset-env', { 'modules': false }]]
        }
    }
}

细粒度导出 ​

优先使用命名导出 (export const a = 1)，避免默认导出大对象。

javascript
export const a = 1;
export const b = 2;

// 不推荐
export default { a: 1, b: 2, c: 3 };

在异步模块中使用 Tree-Shaking ​

Webpack5 之后，可以用一种特殊的备注语法，来实现异步模块的 Tree-Shaking 功能。

通过 /* webpackExports: xxx */ 备注语句，显式声明异步模块中会被使用的内容。

javascript
import(/* webpackExports: ["foo", "default"] */ "./foo").then((module) => {
  console.log(module.foo);
});

Source Map ​

Source Map 是一种映射关系，它将经过压缩、混淆、合并后的产物代码，还原回未打包的原始形态。

其主要作用是帮助开发者在浏览器控制台（DevTools）中，精确定位错误发生的文件、行、列位置，从而极大提升调试效率。

核心原理 ​

Source Map 的实现依赖于 .map 文件（如 bundle.js.map），该文件是一个 JSON 对象，存储了原始代码与产物代码之间的映射关系。

当浏览器开启 Source Map 功能（默认开启）并加载到包含 //# sourceMappingURL=xxx.map 注释的文件时，会自动请求对应的 Map 文件，还原源码。

Map 文件的核心字段：

  • sources: 原始文件路径列表。
  • names: 原始变量名列表。
  • mappings: 记录位置映射关系的字符串（使用 VLQ 编码）。
  • sourcesContent: 原始代码内容。

Webpack 配置 (devtool) ​

Webpack 提供了 devtool 配置项来控制 Source Map 的生成。它支持 20 多种模式，但实际上是由几个核心关键字组合而成的。

关键字描述
source-map生成独立的 .map 文件。
eval每个模块用 eval() 执行，Source Map 作为 DataURL 附加。构建速度最快。
inline将 .map 内容通过 Base64 DataURL 内联到 JS 文件中。不生成独立文件。
cheap只包含行信息，不包含列信息。构建速度较快。
module包含 Loader 处理之前的源代码（如写的 ES6/React/Vue 代码），而不仅仅是 Loader 处理后的代码。
hidden生成 .map 文件，但在 JS 文件中不包含 //# sourceMappingURL 指令。浏览器不会自动加载。
nosources.map 文件中不包含源码内容 (sourcesContent)，只包含映射关系。

所有复杂的配置值都是上述关键字的组合。如

  • cheap-source-map: 有 map 文件，只有行信息。
  • inline-source-map: 内联 map，有行列信息。
  • eval-cheap-module-source-map: 使用 eval 执行，只有行信息，包含 Loader 前源码。

最佳实践 ​

  • 开发环境 (Development)：eval-cheap-module-source-map

eval: 构建和热更新速度快。
cheap: 忽略列信息（通常调试只需要行号），进一步提升速度。
module: 可以看到原本写的代码（如 Vue/React 源码），而不是 Babel 转译后的代码。

  • 生产环境 (Production)：hidden-source-map 或 nosources-source-map

不要直接使用 source-map：会直接暴露源码，安全性低。
hidden-source-map: 生成 map 文件但不自动加载。适合配合 Sentry 等监控平台上传 map 文件，用于线上错误堆栈还原，但用户无法直接查看源码。
nosources-source-map: 会在控制台显示源码目录结构，但文件内容为空。既能看到错误出现的文件路径，又保护了代码内容。
